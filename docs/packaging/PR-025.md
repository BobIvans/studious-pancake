# PR-025 — Packaging, dependency boundaries and working container

## Contract

PR-025 creates one installable, reproducible runtime boundary without claiming a
working trading vertical. CPython `3.13.x` is the only supported interpreter.
The installed executable is `flashloan-bot`; root `arb_bot.py` is a temporary
compatibility wrapper.

The container default is a deliberately network-free `disabled` safe-idle
supervisor. It validates the capability contract and updates an atomic heartbeat
file. It starts no detector, provider, RPC client, simulator, signer or sender.

## Dependency profiles

`pyproject.toml` is the only source of direct dependencies.

- `requirements.txt`: production runtime only.
- `requirements-analytics.txt`: runtime plus offline analytics/ML.
- `requirements-dev.txt`: runtime, analytics, service adapters and quality/test
  tooling.
- `requirements-data.txt`: compatibility alias to the analytics lock.

Heavy analytics packages and HTTP service tooling are not installed into the
runtime image. All direct dependencies are exact pins. Transitive resolutions
are frozen by `uv==0.10.0` for CPython 3.13 on Linux/macOS.

## Updating locks

1. Use CPython 3.13 in a clean virtual environment.
2. Install the current `requirements-dev.txt` or at least the exact resolver
   declared in `pyproject.toml`.
3. Edit only `pyproject.toml` direct dependencies.
4. Run `python scripts/lock_requirements.py` (`make lock`).
5. Review all three lock diffs and `config/requirements-lock.json` hashes.
6. Install each relevant profile in a clean environment and run `pip check`.
7. Run `make verify`, `make package-smoke`, and `make image-smoke`.
8. Commit `pyproject.toml`, all changed locks and the hash manifest together.

The lock script intentionally resolves runtime and analytics directly, resolves
service/dev components independently, then merges exact pins. It fails on an
unowned transitive-version conflict. Direct exact pins win only when the package
is explicitly owned by one project dependency group.

## Wheel smoke

`scripts/package_smoke.py` builds a wheel, inspects required package resources
and console entrypoints, installs it into a temporary environment, changes the
working directory outside the repository, then executes `status` and
`capabilities`. This prevents an accidentally successful smoke caused by imports
from the checkout.

## Container design

The Dockerfile uses `python:3.13.13-slim-bookworm` in a builder and runtime
stage. The builder installs the runtime lock and package. The final image copies
only the virtual environment and contains no compiler, source checkout,
analytics lock, test suite or research datasets.

The final process runs as UID/GID `10001`. `/run/flashloan-bot` is the only
runtime-owned directory introduced here. The healthcheck calls the installed
`flashloan-bot-healthcheck` executable and validates:

- expected heartbeat schema;
- `disabled` mode;
- live PID;
- bounded heartbeat age.

This is process liveness only. It must not be used to claim RPC, provider,
market, paper, shadow or live readiness. PR-042 owns dependency-aware HTTP
`/health`, `/ready`, observability and SLO semantics.

## Rollback

Reverting PR-025 restores script-based execution and the previous container.
No database migration, on-chain state, secret format or live capability is
introduced. Existing callers may continue using `python arb_bot.py` during the
rollback window because the wrapper remains in this PR.
