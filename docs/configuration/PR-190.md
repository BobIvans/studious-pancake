# PR-190 — Canonical configuration ingestion and identity-preserving activation

## Security boundary

This change makes configuration bytes unambiguous before they can influence
readiness, operator confirmation, policy hashing, or permit issuance.

The active runtime and limited-live policy paths now share these invariants:

- duplicate YAML keys are fatal at every depth;
- anchors, aliases, merge keys, non-string mapping keys, implicit timestamps,
  oversized documents and excessive nesting are rejected;
- the limited-live policy is a versioned Pydantic model with unknown-field and
  strict scalar rejection;
- safe display, identity payload and runtime materialization are separate APIs;
- secret values never enter a policy identity, while secret scheme, locator,
  optional version and usage scope remain identity-bearing;
- canonical hashes use a domain-, schema- and environment-separated JSON envelope;
- source manifests and effective-config diffs contain hashes/provenance rather than
  secret values;
- activation records are signed, immutable, atomically replaced and protected by
  compare-and-swap against the reviewed previous generation.

## Compatibility

`LiveRiskPolicy` keeps a mapping surface so existing readiness/admission code and
focused PR-018 tests continue to use `.get()`, indexing and validated assignment.
The existing live-risk schema version remains `pr018.live-risk.v1`; PR-190 hardens
its ingestion and identity semantics without silently changing financial limits.

## Activation protocol

`ConfigActivationStore.activate()` creates a monotonically increasing signed
record bound to the policy hash, release hash, source-manifest hash, approvals,
creation time and expiry. Activation fails when the reviewer-approved previous
generation differs from the currently active generation. Writes use a process
lock, owner-only temporary file, `fsync`, atomic `os.replace`, directory `fsync`,
and symlink rejection.

## Verification

```bash
python -m pytest -q \
  tests/test_pr026_typed_config.py \
  tests/test_live_readiness.py \
  tests/test_pr190_canonical_config.py
python -m compileall -q src
```

Negative coverage includes duplicate live-enable/runtime keys, YAML merge and
alias use, non-string keys, unknown live-policy fields, distinct secret locators,
stale activation approval and identity/financial effective-config changes.

## Deliberate exclusions

This PR does not resolve secret values, enable live trading, issue a real signing
permit, or change the fail-closed default. Deployment-specific signature key
provisioning and binding a production release pipeline to the activation store
remain operator integration work and must not be inferred from the presence of
these primitives.
