# PR-201 — Cryptographic nonce generation and authorization replay protection

## Scope

PR-201 replaces hash-shape-only nonce acceptance with a trusted, domain-separated,
durable one-time authorization boundary. The module is offline and standard-library
only. It does not load a private key, sign a transaction, call RPC, or enable live
submission.

## Threat model closed

The previous PR-141 structural primitive accepted any lowercase 64-character value,
including an all-zero digest. Changing only `authorization_id` could authorize the
same exact message again. PR-201 closes that boundary by requiring a challenge that:

- is generated by one reviewed CSPRNG-backed issuer;
- contains at least 256 bits of nonce material;
- is MAC-bound to issuer/key version, purpose, domain, release, PolicyBundle,
  attempt generation, exact operation hash, environment, cluster and expiry;
- is inserted into a durable SQLite replay ledger before use;
- has a globally unique nonce digest, so a nonce cannot cross domains;
- transitions through `issued -> reserved -> consumed` exactly once;
- stores the exact result and returns it for an identical retry without invoking the
  backend again.

Obvious all-zero, constant-byte and repeated-pattern nonce material is rejected.
The legacy PR-141 primitive also rejects placeholder-shaped SHA-256 digests, but
production callers must use `authorize_pr141_once()` rather than direct structural
authorization.

## Crash and concurrency semantics

PR-201 deliberately does not claim that a generic signer callback can be made atomic
with SQLite across a process crash. Instead it uses a fail-closed fenced protocol:

1. reserve the durable challenge and deterministic operation ID;
2. call an idempotent backend using that operation ID;
3. atomically commit `consumed + exact result` before returning success.

If the backend outcome becomes unknown, the challenge remains `reserved`. A retry
never calls the backend again blindly. An operator or signer adapter must reconcile
by operation ID and call `recover_result()` with the retrieved original result, or
revoke/conflict the challenge. This prevents duplicate signing while preserving an
honest unknown-outcome state.

Concurrent callers race on `BEGIN IMMEDIATE` plus a state compare-and-swap. Only one
caller obtains the reservation; others receive `in_progress` or a replay conflict.

## Database contract

The SQLite file carries an immutable product/schema identity and an explicit busy
timeout. It must not share a path with another database product. The synchronous
ledger is intended for an isolated signer/operator process or a PR-200-owned worker;
it must not be called directly on the main asyncio event loop.

## Integration boundary

`authorize_pr141_once()` binds the PR-141 request to the issued PR-201 challenge:

- authorization ID equals the server-generated challenge ID;
- attempt ID/generation and PolicyBundle match;
- the exact unsigned message hash is the challenge operation hash;
- the nonce digest comes from the trusted issuer record;
- the durable result is reconstructed on exact retry without reauthorization.

The same replay service supports separate domains for signer transactions, operator
approvals, recovery actions, release activation, submission permits and treasury
transfers. Domains share the ledger framework but never share authorization meaning.

## Non-goals

- no private-key or wallet import;
- no transaction signing or submission;
- no live enablement;
- no automatic recovery from an unknown signer outcome;
- no replacement for PR-183 credential authority or PR-200 async persistence
  ownership;
- no claim that process-local legacy permit issuers are already cut over.

## Verification

```bash
python -m pytest tests/test_pr201_authorization_replay.py -q
python -m py_compile \
  src/authorization_replay_pr201.py \
  src/signer_authorization_pr141.py \
  tests/test_pr201_authorization_replay.py
python -m compileall -q src tests
```

Focused tests cover all-zero nonce rejection, CSPRNG issuance, placeholder rejection,
MAC/domain tampering, cross-domain nonce reuse, single-use PR-141 authorization,
idempotent exact retry, conflicting second envelope, expiry/revocation, unknown
backend outcome, restart recovery and concurrent execution.
